### 1. 完成 warm start 第一版接入
今天明确了当前 warm start 的设计口径：

- 只在初始时做一次 warm start
- 不在每一轮 receding horizon 中重复生成 warm start
- 当前 warm start 的形式是：使用 goal-directed 的初始控制序列替代 baseline 的固定 nominal sequence 初始化

同时，代码中已经保留 `use_goal_warm_start` 开关，能够直接切换 baseline 与 warm start 模式。

### 4. 新增 top-k 预测轨迹可视化
为了判断问题到底出在：

- 候选轨迹没有采到正确路线
- 还是采到了但没有被选中

今天新增了 top-k 预测轨迹可视化，包括：

- 第 0 步 top-k predicted trajectories
- 最后一轮 top-k predicted trajectories
- 最后一轮 best predicted trajectory

这一步帮助更清楚地区分：

- 初始候选簇是否已经偏向错误 corridor
- 最终 best trajectory 是否本身就落在错误同伦类上

---

### 5. 重新定位当前 dense 场景的核心问题
今天的重要结论之一是：

当前 dense 场景里的主要问题，不只是执行轨迹没有跟着某条“正确路径”走，而是规划器在某些设置下，连 best predicted trajectory 本身就可能落在错误 corridor 上。

也就是说，当前问题已经不仅仅是执行层的问题，而是：

- 采样候选簇
- 代价排序
- nominal sequence 更新
- horizon 长度

共同作用下导致的 route 选择偏差。

---

### 6. 发现 dense 场景对 horizon 非常敏感
今天非常明确地观察到：

- 在 dense 场景中，baseline 在较小 horizon 下，反而更容易保持在正确局部 corridor
- horizon 变大后，更容易被带到不理想的全局路线
- 因此，当前现象更适合解释为 **long-horizon bias**，而不是“过拟合”

也就是说：

- 长 horizon 并没有自动带来更好的规划结果
- 在当前实现下，较长 horizon 反而更容易把候选轨迹簇推向错误同伦类
- dense 场景目前是对 horizon 最敏感的场景

---

### 7. 对 warm start 与 horizon 的作用做了区分
今天后半段也进一步澄清了两条实验线：

#### horizon 的作用
主要体现在：
- baseline 本身更容易落在哪一类 corridor

#### warm start 的作用
主要体现在：
- 在固定 horizon 下，是否改变第 0 步候选轨迹簇的初始分布

因此，后续分析时必须把：

- horizon 效应
- warm start 开/关效应

分开讨论，避免把两个变量混在一起解释。

## 今日最关键结论

### 结论 1
one-shot warm start 已经成功接入并可以稳定运行，但其收益目前还不明确。

### 结论 2
dense 场景中的路线错误，并不只是执行阶段的问题，规划阶段本身就可能已经落入错误同伦类。

### 结论 3
dense 场景对 horizon 非常敏感，当前实现中较小 horizon 反而更容易保持在正确 corridor；较长 horizon 更容易表现出 long-horizon bias。

### 结论 4
top-k 预测轨迹可视化是有效调试工具，能够帮助区分“没采到正确路”和“采到了但没选中”这两类问题。

---

## 当前仍未解决的问题

1. warm start 在 dense 场景中的实际增益仍不明确  
2. 长 horizon 下为什么更容易偏向错误 corridor，还需要更系统的实验支持  
3. 需要继续用表格和指标而不是只靠图来整理 dense 场景的 horizon 效应  
4. horizon 与 warm start 这两条实验线需要彻底拆开分析
